# React Query에서 쿼리 추상화 만들기

> 이 글은 TkDodo의 [Creating Query Abstractions](https://tkdodo.eu/blog/creating-query-abstractions)를 번역한 글입니다.

![Creating Query Abstractions](/thumbnail/creating-query-abstractions/index.jpg)

개발자들은 추상화를 참 좋아합니다.

다른 곳에서도 쓸 것 같은 코드가 보이면 추상화를 만들고, 고작 3줄짜리 코드가 살짝 달라도 플래그를 추가해 추상화를 만들고,

모든 `useQuery` 에 공통으로 뭔가 적용하고 싶어지면 또 추상화를 만듭니다. 그렇게 추상화는 계속 쌓여가고, 결국 모든 코드가 추상화 위에 쌓여가는 것이죠.

추상화 자체가 나쁜 건 아닙니다. 하지만 모든 것이 그렇듯 트레이드오프가 있습니다. Dan의 강연 [The Wet Codebase](https://www.deconstructconf.com/2019/dan-abramov-the-wet-codebase)는 제가 가장 좋아하는 강연 중 하나인데, 이 점을 정말 잘 설명해 줍니다.

## 커스텀 훅

React에서 추상화를 만드는 것은 대부분 커스텀 훅과 연결됩니다. 여러 컴포넌트 간 로직을 공유하거나, 복잡한 `useEffect`를 의미 있는 이름 뒤에 숨기는 데 아주 유용하죠. 오랫동안 `useQuery` 위에 커스텀 추상화를 만드는 방법은 커스텀 훅을 작성하는 것이었습니다:

```ts
function useInvoice(id: number): UseQueryResult<Invoice, Error> {
  return useQuery({
    queryKey: ['invoice', id],
    queryFn: () => fetchInvoice(id),
  });
}
```

이 방식은 `queryKey`와 `queryFn`을 하나의 위치에 묶어 주고, 반환 타입도 명시적이므로 나쁘지 않습니다. 하지만 문제는 **유연성이 부족하다**는 것입니다.

`useQuery`에는 24개의 옵션이 있는데, 현재 추상화로는 그것들을 전달할 수 없습니다. 특정 화면에서 백그라운드 업데이트가 중요하지 않아 다른 `staleTime`을 주고 싶다면? 파라미터로 받으면 되겠죠:

```ts
function useInvoice(id: number, staleTime: number) {
  return useQuery({
    queryKey: ['invoice', id],
    queryFn: () => fetchInvoice(id),
    staleTime,
  });
}
```

그런데 이번엔 누군가 Error Boundary 연동을 위해 `throwOnError`도 넘기고 싶어 합니다. 파라미터가 너무 많아지니 처음부터 객체로 받을걸 싶어지죠:

```ts
function useInvoice(id: number, options?: { staleTime?: number; throwOnError?: boolean }) {
  return useQuery({
    queryKey: ['invoice', id],
    queryFn: () => fetchInvoice(id),
    ...options,
  });
}
```

이쯤 되면 제대로 가고 있는 건지 의문이 생깁니다. 새로운 사용 사례가 생길 때마다 이 작은 추상화를 계속 건드려야 하는 건 이상적이지 않습니다. 반환값은 라이브러리가 반환하는 것을 그대로 따르기로 했는데, 옵션도 그렇게 할 수 없을까요?

## UseQueryOptions 시도

React Query가 `UseQueryOptions` 타입을 제공하니까, 옵션 전체를 받아서 스프레드하면 될 것 같습니다:

```ts
function useInvoice(id: number, options?: Partial<UseQueryOptions>) {
  return useQuery({
    queryKey: ['invoice', id],
    queryFn: () => fetchInvoice(id),
    ...options,
  });
}
```

하지만 타입 파라미터를 지정하지 않으면 기본값이 사용되는데, `data`의 기본값이 `unknown`이라 그게 퍼져버립니다.

### 잘못된 해결책 — 제네릭 직접 지정

흔히 보이는 어리숙한 해결책은 `UseQueryOptions`에 첫 번째 타입 파라미터를 직접 선언하는 것입니다:

```ts
function useInvoice(id: number, options?: Partial<UseQueryOptions<Invoice>>) {
  return useQuery({
    queryKey: ['invoice', id],
    queryFn: () => fetchInvoice(id),
    ...options,
  });
}
```

`data` 타입은 다시 살아나지만, `select` 같은 옵션에서 무너집니다:

```ts
const { data } = useInvoice(1, {
  select: (invoice) => invoice.createdAt,
  // ❌ 오류: Type '(invoice: Invoice) => string' is not assignable to type
  //         '(data: Invoice) => Invoice'
});
```

더 많은 타입 파라미터를 추가할 수도 있지만, 그러면 "그냥 JavaScript처럼 보이는 코드"라는 약속에서 점점 멀어집니다.

## 커스텀 훅의 또 다른 문제

타입 문제 외에도, 커스텀 훅에는 근본적인 한계가 있습니다:

- **훅은 컴포넌트나 다른 훅 안에서만 호출할 수 있습니다.** 하지만 쿼리 설정은 라우트 로더, 서버 컴포넌트, 이벤트 핸들러 등 다양한 곳에서 필요합니다.
- **특정 구현에 종속됩니다.** `useQuery`로 감싸면 `useSuspenseQuery`를 쓰고 싶을 때 새로운 훅을 만들어야 합니다.
- **queryClient 메서드와 함께 쓸 수 없습니다.** `queryClient.prefetchQuery`나 `queryClient.invalidateQueries`에는 훅이 아닌 옵션 객체가 필요합니다.

## 더 나은 추상화: queryOptions

저는 커스텀 훅이 여기서 적합한 추상화가 아니라는 결론에 도달했습니다. 대신, React Query v5에서 제공하는 `queryOptions` 함수를 추천합니다.

`queryOptions`는 다양한 훅 간에, 심지어 명령형 함수와도 공유할 수 있습니다. 그냥 일반 함수이기 때문에 어디서든 동작합니다. 런타임에서는 아무것도 하지 않습니다. 트랜스파일된 결과를 보면:

```js
// queryOptions.js
function queryOptions(options) {
  return options;
}
```

하지만 타입 레벨에서는 강력한 힘을 발휘합니다:

```ts
import { queryOptions } from '@tanstack/react-query';

function invoiceOptions(id: number) {
  return queryOptions({
    queryKey: ['invoice', id],
    queryFn: () => fetchInvoice(id),
  });
}

const { data: invoice1 } = useQuery(invoiceOptions(1));
// data: Invoice | undefined

const { data: invoice2 } = useSuspenseQuery(invoiceOptions(2));
// data: Invoice  ✅ (undefined 없음!)
```

어떤 훅과 함께 쓰느냐에 따라 타입이 자동으로 추론됩니다. 물론 `queryClient` 메서드와도 완벽하게 동작합니다:

```ts
// 프리페칭
queryClient.prefetchQuery(invoiceOptions(1));

// 캐시 무효화
queryClient.invalidateQueries({
  queryKey: invoiceOptions(1).queryKey,
});
```

## QueryOptions 합성하기

그럼 옵션은 어떻게 전달하냐고요? 사실 그럴 필요가 없습니다. `invoiceOptions`는 모든 사용처에서 공유하고 싶은 옵션만 담으면 됩니다. 추가 옵션이 필요하면 사용하는 곳에서 스프레드로 합성하면 됩니다:

```ts
const { data } = useQuery({
  ...invoiceOptions(1),
  throwOnError: true,
  select: (invoice) => invoice.createdAt,
});
```

이 방식은 완전히 타입 안전하고, 완전히 직관적입니다.

물론 여전히 커스텀 훅을 만들 수 있지만, 그것들은 `queryOptions` 위에 구축되어야 합니다. `queryOptions`가 가장 먼저 손을 뻗어야 할 첫 번째 추상화 빌딩 블록이니까요. **추상화에선 단순함이 최고인데** 이보다 더 단순할 수가 없습니다. 👑
